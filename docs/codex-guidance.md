# Подсказки для Codex/LLM при работе с проектом

## Нужны ли дополнительные описания
- Текущие документы покрывают цепочку сервисов (бот → API базы знаний → Document Processor → эмбеддер → поиск) и настройки портов. Этого достаточно для навигации по проекту.
- Дополнительные описания не требуются для базовой работы Codex, но небольшие уточнения ускорят ответ: чёткие входы/выходы API, примеры запросов и ожидаемые статусы обработки.

## Что помогает модели работать точнее
- **Контракты API**: краткие спецификации эндпоинтов (параметры, коды ответа, примеры), особенно для `/v1/answer`, `/items/{id}` и очередей Document Processor.
- **Конфигурация**: список переменных окружения с типами/дефолтами (`embedding_config`, `OCR_BACKEND`, порты сервисов), чтобы не искать их по коду.
- **Пайплайн обработки**: схема событий/статусов (upload → enrichment → deletion) и правила версионирования эмбеддингов.
- **Тестовые сценарии**: примеры входных файлов для парсеров (PDF/DOCX) и запросов для поиска, чтобы быстро проверить изменения.
- **Ошибки и fallback**: описание резервных путей (LLM/Оллама, OCR EasyOCR/Tesseract), чтобы понимать поведение при сбоях.

## Как добавлять полезные описания
- Короткие `.md` файлы в `docs/` или рядом с модулем с перечислением контрактов и конфигов.
- Минимальные таблицы статусов/переходов для Document Processor и унифицированного эмбеддера.
- В README сервисов — раздел «Быстрый старт» с командами запуска и проверками здоровья.

Эти дополнения не обязательны, но сократят время навигации и уточнения контекста для Codex/LLM.
